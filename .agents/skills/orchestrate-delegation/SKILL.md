# orchestrate-delegation — 委任オーケストレーションの実行手順

worker/subagent への委任は、オーケストレーターのアドホック判断に頼ると事故る。典型的な失敗は、worker の独断行動・指示無視、同一スコープの重複起動、無承認の大規模並列、そして完了報告の鵜呑みである。これらは委任の各段階に検証ポイントを外部化することで防げる。

委任は以下の 7 ステップを**順番に、飛ばさず**実行する。各ステップは判断不要なレベルまで具体化してある。ブリーフ文面は `worker-brief` スキルを参照する。

## ステップ 1: スコープ明確化

- タスクを **1 worker = 1 明確スコープ**に分割する。「いい感じに」「全体を直す」等の曖昧タスクはここで具体化する。
- 各サブタスクについて「変更してよいファイル/範囲」と「やらないこと」を書き出す。
- **エスカレーション判定**: 以下のいずれかに当たるなら、委任せずユーザーに確認する。
  - スコープが一意に定まらない
  - 破壊的・不可逆操作（本番反映・データ削除・共有ブランチへのマージ等）を含む
  - 受け入れ条件を機械検証可能な形で書けない

## ステップ 2: 重複起動チェック（検証可能な形で）

- **spawn 直前に、稼働中の worker 一覧（tmux ペイン・エージェントのチーム/セッション一覧等、環境で使える手段）を確認**し、同一スコープの worker が存在しないことを確かめる。
- 確認結果を一行で報告する（例: 「tmux 3 ペイン / 既存 worker に同一スコープなし → 新規 spawn 可」）。
- 同一スコープの worker が既にあれば新規起動せず、既存を使う。
- **agmsg identity（actas/claim）はメインプロジェクトパス基準で登録される**。worktree cwd から叩くと `not_registered`（Exit 2）になり得るので、identity 操作にはメインパスを渡す。

### チーム/worker 設定に関わる確認質問を出す前に

- ユーザーに確認質問（AskUserQuestion）で選択肢を組む前に、**既存のチーム/worker 一覧と tmux ペインを確認**する。
- 選択肢には**必ず「新規作成」を含める**。既存チーム/worker 前提の選択肢だけを出さない（ユーザーの割り込み訂正を招く）。

## ステップ 3: 受け入れ条件の定義（委任前に決める）

各サブタスクについて、報告時に**機械的に照合できる**受け入れ条件を定義する。以下から該当するものを選び具体値を書く:

- **diff 範囲**: 変更がスコープ内ファイルに収まる（`git diff --stat` で確認）。範囲外 diff があれば不合格。
- **テスト**: 指定テストコマンドがグリーン。
- **ビルド/型チェック**: 通過。
- **視覚検証**（UI）: デザイン基準（デザインツールの数値・スクリーンショット等）との照合で、事前に合意した一致度基準を満たす。
- **禁止事項**: 勝手なアイコン/装飾/文言の追加が diff に無い。

## ステップ 4: モデル選択

作業性質に応じてモデルの能力ティアを割り当てる:

| 作業の性質 | ティア | 補足 |
|---|---|---|
| レビュー・検証・受け入れ判定・計画立案 | 上位（例: opus） | 判断・見落とし検出が要る工程。検証ゲートの照合役もここ。 |
| 実装（スコープ・受け入れ条件が明確） | 中位（例: sonnet） | `worker-brief` の厳格ブリーフ必須。曖昧なまま投げない。 |
| 機械的作業（一括置換・整形・定型変換・収集） | 下位（例: haiku） | 判断不要な反復作業。 |

- 中位モデルに実装を委任する条件は、スコープ・やらないこと・受け入れ条件が全て明文化済みであること。曖昧なら上位モデルに格上げするか、自分で分割し直す。
- **検証・レビュー工程を実装と同じ worker に兼任させない**（自己検証は甘くなる）。検証は別途、上位モデル（またはオーケストレーター自身）で行う。

## ステップ 5: ブリーフ作成と委任

- `worker-brief` スキルのテンプレートを埋める（スコープ/やらないこと/中断ルール/セルフチェック/報告フォーマット）。ステップ 1〜4 の内容をそのまま流し込む。
- **並列度は同時 2〜3 まで**。超える場合は起動前に範囲・構成・並列数をユーザーに提示し承認を得る。
- 多段オーケストレーション（実装→レビュー→修正→マージ等）が必要なら、その場で組まず `plan-phase-delivery` / `parallel-phase-delivery` 等の既存スキルに乗せる。
- **spawn 前に worker の worktree へ agmsg の monitor 配信モードを設定する（必須）**: `~/.agents/skills/agmsg/scripts/delivery.sh set monitor claude-code <worktree絶対パス>`。worktree は `.claude/settings.local.json` を持たないため既定で mode `off` になり、worker は agmsg inbox Monitor を起動せず**完了報告後の追加依頼（レビュー対応等）を受け取れない**。worktree を作り直すたびに再設定が必要。
- 委任を実行する。**spawn/worker 起動の readiness を foreground ブロッキングで待たない**。spawn はバックグラウンドで投げるか短タイムアウトで発火だけ確認し、起動完了は Monitor（tmux ペイン state / チーム membership / inbox）で待つ。foreground の長時間ブロックはシェルツールのタイムアウトで SIGTERM される。
- **spawn 後、最初のブリーフ/追加指示を送る前に worker の watcher attach を確認する**: spawn スクリプトの `status=ready` 出力、または `~/.agents/skills/agmsg/scripts/delivery.sh status claude-code <worktree絶対パス>` の watch process で確認。attach していない worker への送信は idle 後に届かない（tmux send-keys での nudge は設定漏れ時の緊急対処にとどめる）。

## ステップ 6: 報告の検証ゲート

worker の完了報告を受けたら、そのまま信用せず**ステップ 3 の受け入れ条件を 1 つずつ照合**する:

- [ ] diff がスコープ内に収まっている（範囲外変更が無い）
- [ ] テスト/ビルド/型チェックが指定どおり通過
- [ ] （UI）視覚検証が合意した基準を満たす
- [ ] 勝手な追加要素が無い
- [ ] 受信メッセージ・停止指示に応答済み

検証は可能な限りオーケストレーター自身で実際にコマンドを走らせて確認する。「報告に書いてある」だけを根拠にしない。

## ステップ 7: 合否判定と差し戻し

- **全条件クリア** → 合格。次のサブタスク or 統合へ進む。
- **未達あり** → 不合格。**具体的な不足点**（どの条件がどう未達か）を添えて worker に差し戻す（再委任）。ステップ 5〜6 を繰り返す。
- 差し戻しを 2〜3 回繰り返しても収束しないときは、スコープ分割かモデル格上げ（中位→上位）を見直し、必要ならユーザーにエスカレーションする。
