# コミットメッセージの例

Good（良い）例と Bad（悪い）例を通じて、効果的なコミットメッセージの書き方を学びます。

## ✅ Good Examples

### 例1: 新機能追加（認証）

```
feat(auth): JWT検証にキャッシュを導入

JWT の公開鍵取得APIは頻繁に呼ばれるため、
1時間のキャッシュを導入してレスポンス時間を改善。

公開鍵は通常変更されないため、短時間のキャッシュで
セキュリティリスクを最小化しつつパフォーマンスを向上。
```

**良い点:**
- Type と Scope が明確
- Subject が WHY を暗示（「高速化」が目的）
- Body で変更の理由と背景を説明
- 技術的判断の根拠を明記

---

### 例2: バグ修正（金額計算）

```
fix(billing): 金額計算で小数点以下が切り捨てられる問題を修正

固定小数点（10進）演算を使用することで、浮動小数点演算の誤差を
回避し、金額を正確に計算できるようにする。

float型の演算では 0.1 + 0.2 = 0.30000000000000004 のような
誤差が発生し、金額計算では許容できない。10進演算は
正確な結果を保証する。

Fixes #234
```

**良い点:**
- Bug の原因（浮動小数点演算の誤差）を明記
- 解決策（10進演算）の選択理由を説明
- 技術的背景を詳しく記載
- Issue 番号でトレーサビリティを確保

---

### 例3: リファクタリング（Repository層）

```
refactor(repository): 複雑なクエリをRepositoryに移動

Controllerの肥大化を防ぎ、クエリロジックの再利用性と
テスタビリティを向上させるため、Repository層に集約。

現状、同じクエリが複数のControllerで重複しており、
保守性が低下していた。Repository パターンにより、
単一責任の原則に従った設計に改善。
```

**良い点:**
- リファクタリングの目的が明確
- 現状の問題点を説明
- アーキテクチャ設計の意図を明記

---

### 例4: パフォーマンス改善

```
perf(list): 一覧取得のN+1問題を解消

Eager Loadingを使用して、1回のクエリで関連データを取得。
100件表示時のクエリ数を101回から2回に削減し、
レスポンス時間を3秒から200msに改善。
```

**良い点:**
- 具体的な改善数値を記載
- 技術的手法（Eager Loading）を明記
- ビフォー・アフターが明確

---

### 例5: Breaking Change

```
feat(api): ユーザーAPI のレスポンス形式を統一

BREAKING CHANGE: すべてのユーザーAPIレスポンスを data キーで
ラップする形式に変更。

変更前:
{
  "id": 1,
  "name": "Alice"
}

変更後:
{
  "data": {
    "id": 1,
    "name": "Alice"
  }
}

クライアント側でエラーハンドリングとデータ取得を
統一的に扱えるようにするため、レスポンス表現を標準化。

Closes #567
```

**良い点:**
- BREAKING CHANGE であることを明記
- 変更前後の具体例を提示
- 変更の意図を説明

---

## ❌ Bad Examples

### 例1: 情報不足

```
fix: バグ修正
```

**問題点:**
- Scope がない（どこのバグ？）
- どんなバグを修正したのか不明
- WHY が全くない

**改善例:**
```
fix(billing): 金額計算の小数点誤差を修正

浮動小数点演算による誤差を避けるため、10進演算を導入。
```

---

### 例2: WHAT の羅列

```
feat(admin): AdminController.php を変更して、HTTPSチェックを追加し、
リダイレクト機能を実装した
```

**問題点:**
- Subject が長すぎる（50文字超過）
- 「何をしたか」の羅列で「なぜ」がない
- diff を読めば分かる内容

**改善例:**
```
feat(admin): HTTPSリダイレクト機能を追加

セキュリティ強化のため、HTTP接続を自動的にHTTPSへリダイレクト。
```

---

### 例3: 不適切な Type

```
update: コードを修正
```

**問題点:**
- `update` は Conventional Commits の有効な type ではない
- Scope がない
- 何を修正したのか不明

**改善例:**
```
refactor(auth): JWTデコード処理を関数に抽出

コードの可読性と再利用性を向上させるため、
複雑なJWTデコードロジックを独立した関数に分離。
```

---

### 例4: 過去形・現在完了形を使用

```
feat(api): Added new endpoint for user search
```

**問題点:**
- 過去形（Added）は Conventional Commits の規約違反
- Subject は命令形が正しい

**改善例:**
```
feat(api): ユーザー検索エンドポイントを追加

部分一致検索と曖昧検索をサポートし、
管理画面での効率的なユーザー検索を可能にする。
```

---

### 例5: 曖昧な説明

```
fix: いろいろ修正
```

**問題点:**
- 「いろいろ」では何も伝わらない
- Scope がない
- 複数の無関係な変更が含まれている可能性

**改善案:**
- 無関係な変更は別コミットに分割
- 各変更に明確な説明を付ける

---

## 例

### データベースマイグレーション

```
feat(db): sessions テーブルに device_id カラムを追加

デバイス識別のため、デバイス固有IDを保存する必要がある。
users テーブルの id とは別物で、セッションごとの
端末識別に使用。

Refs #789
```

---

### インフラ（IaC）変更

```
feat(infra): Prod環境にロードバランサーを追加

HTTPS通信を終端し、バックエンドへの通信を暗号化するため、
ロードバランサーを導入。
証明書と統合し、TLS 1.3 を有効化。

Closes #122
```

---

### API スキル修正

```
refactor(skills): git-commit スキルをProgressive Disclosure化

SKILL.mdを簡素化し、詳細をreferences/に分割することで、
コンテキストウィンドウの効率を改善。必要な情報のみを
段階的に読み込めるようにする。
```

---

### テスト追加

```
test(billing): 金額計算の境界値テストを追加

ゼロ、上限値、小数点以下の端数処理など、
エッジケースでの動作を保証するテストを実装。
```

---

## ステージング戦略の例

### 単一ファイルの一部のみコミット

```bash
# インタラクティブモードで選択
git add -p src/services/order-service.*

# Hunk ごとに y/n で選択
# y: このHunkをステージング
# n: スキップ
# s: さらに小さく分割
# q: 終了
```

**コミットメッセージ例:**
```
refactor(order): 合計金額の算出ロジックを独立した関数に抽出

再利用性とテスタビリティを向上させるため、
calculateTotal() 関数を作成。
```

残りの変更は別コミット:
```
feat(order): 割引適用機能を追加

数量割引・クーポン割引・会員割引の適用をサポート。
```

---

### 複数の無関係な変更を分割

**変更内容:**
- `src/services/auth-service.*`: JWT検証の改善
- `src/services/order-service.*`: 金額計算のバグ修正
- `README.md`: ドキュメント更新

**コミット1:**
```bash
git add src/services/auth-service.*
git commit -m "feat(auth): JWT検証にキャッシュを導入"
```

**コミット2:**
```bash
git add src/services/order-service.*
git commit -m "fix(billing): 金額計算の小数点誤差を修正"
```

**コミット3:**
```bash
git add README.md
git commit -m "docs: セットアップ手順を更新"
```
