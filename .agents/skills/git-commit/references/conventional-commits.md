# Conventional Commits 詳細仕様

Conventional Commits は、コミットメッセージに人間と機械が読みやすい意味を持たせるための仕様です。

## フォーマット

```
<type>(<scope>): <subject>

<body>

<footer>
```

## Type（必須）

変更の種類を示します：

| Type | 説明 | 使用例 |
|------|------|--------|
| `feat` | 新機能の追加 | 新しいAPI、UI要素、機能 |
| `fix` | バグ修正 | 不具合の解消 |
| `refactor` | リファクタリング | 機能変更なし、コード構造の改善 |
| `docs` | ドキュメントのみの変更 | README、コメント、API仕様書 |
| `style` | コードの意味に影響しない変更 | フォーマット、セミコロン、空白 |
| `test` | テストの追加・修正 | ユニットテスト、統合テスト |
| `chore` | ビルドプロセスや補助ツール | 依存関係更新、設定ファイル |
| `perf` | パフォーマンス改善 | 速度最適化、メモリ削減 |
| `ci` | CI設定ファイルの変更 | GitHub Actions、CircleCI |
| `revert` | 以前のコミットを戻す | 問題のあるコミットの取り消し |

## Scope（推奨）

変更の影響範囲を示します。プロジェクト固有のモジュール名やコンポーネント名を使用：

**例:**
- `auth` - 認証関連
- `admin` - 管理画面
- `api` - API エンドポイント
- `db` - データベース・マイグレーション
- `infra` - インフラコード
- `<feature>` - 各機能ドメインごとに定義
- `skills` - Agent Skills

**省略可能:** 影響範囲が明確でない、または複数に跨る場合

## Subject（必須）

変更の簡潔な説明：

### ルール
- **命令形を使う**: "add" not "added" or "adds"
- **50文字以内**: 簡潔に
- **先頭は小文字**: 例外として固有名詞は大文字
- **末尾にピリオド不要**
- **WHY を暗示する表現を使う**: 単なる変更内容ではなく、目的や理由を示唆

### Good Subject Examples
```
feat(auth): キャッシュを導入してJWT検証を高速化
fix(billing): 小数点演算の誤差を解消するため固定小数点演算を使用
refactor(repository): クエリロジックを再利用可能にするため分離
```

### Bad Subject Examples
```
feat(auth): JWTにキャッシュを追加した
fix(billing): 演算方式に変更
refactor: コードを変更
```

## Body（推奨）

変更の「理由」と「背景」を詳しく説明：

### 含めるべき内容
- **WHY（なぜ）**: なぜこの変更が必要か
- **背景**: どんな問題があったか
- **影響**: どのような効果があるか
- **代替案の検討**: なぜこの方法を選んだか

### 含めるべきでない内容
- **WHAT（何を）**: diff で既に分かる
- **HOW（どうやって）**: コード自体が説明している

### フォーマット
- 空行を挟んで subject と分離
- 72文字で改行
- 複数段落可能

### Body 例
```
管理画面へのHTTPアクセスを自動的にHTTPSにリダイレクトすることで、
セキュリティを強化し、証明書エラーを防ぐ。

本番環境のロードバランサーはHTTPSのみ受け付けるが、開発時に誤ってHTTPで
アクセスするケースがあり、ユーザー体験を損なっていた。
```

## Footer（条件付き）

特殊な情報を記載：

### Breaking Change（互換性を破壊する変更）
```
BREAKING CHANGE: 認証APIのレスポンス形式を変更

tokenフィールドがaccessTokenに改名されました。
クライアントの更新が必要です。
```

### Issue 参照
```
Fixes #123
Closes #456
Refs #789
```

複数のIssueを参照可能：
```
Fixes #123, #456
```

### その他のメタ情報
```
Reviewed-by: @username
Co-authored-by: Name <email@example.com>
```

## 完全な例

```
feat(admin): HTTPSリダイレクト機能を追加

管理画面へのHTTPアクセスを自動的にHTTPSにリダイレクトすることで、
セキュリティを強化し、証明書エラーを防ぐ。

本番環境のロードバランサーはHTTPSのみ受け付けるが、開発時に誤ってHTTPで
アクセスするケースがあり、ユーザー体験を損なっていた。

ミドルウェアでリクエストのスキームをチェックし、HTTPの場合は
301リダイレクトを返す実装とした。

Closes #122
```

## Breaking Change の判断基準

以下の場合は `BREAKING CHANGE:` を含めること：

1. **APIの変更**
   - エンドポイントのURL変更
   - リクエスト/レスポンス形式の変更
   - 必須パラメータの追加

2. **データ構造の変更**
   - データベーススキーマの非互換な変更
   - 設定ファイル形式の変更

3. **動作の大幅な変更**
   - デフォルト動作の変更
   - 削除された機能

4. **依存関係の大幅な更新**
   - メジャーバージョンアップ
   - 非推奨APIの削除
