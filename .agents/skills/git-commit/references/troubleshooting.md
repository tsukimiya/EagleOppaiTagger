# トラブルシューティング

コミット時によく遭遇する問題とその解決策をまとめています。

## Common Pitfalls（よくある落とし穴）

### 1. 無関係な変更が混在している

**問題:**
```bash
git status
# 表示:
# modified:   src/services/auth-service.*    (JWT改善)
# modified:   src/services/order-service.*   (バグ修正)
# modified:   README.md                      (ドキュメント)
```

異なる目的の変更を1つのコミットにまとめると、レビューが困難になり、将来的に特定の変更をrevertする際に問題が発生します。

**解決策:**
```bash
# 変更ごとに個別にステージング
git add src/services/auth-service.*
git commit -m "feat(auth): JWT検証にキャッシュを導入"

git add src/services/order-service.*
git commit -m "fix(billing): 金額計算の誤差を修正"

git add README.md
git commit -m "docs: セットアップ手順を更新"
```

---

### 2. コミットメッセージが WHAT の羅列

**問題:**
```
feat(admin): AdminController.phpにHTTPSチェック処理を追加し、
リダイレクトロジックを実装し、ミドルウェアを登録した
```

diff を見れば「何をしたか」は分かります。重要なのは「なぜその変更が必要だったか」です。

**解決策:**
```
feat(admin): HTTPSリダイレクト機能を追加

HTTP接続時のセキュリティリスクを軽減し、
証明書エラーを防ぐためHTTPS強制を実装。
```

**原則:** コミットメッセージは未来の自分や他の開発者が「この変更の意図」を理解するためのもの。

---

### 3. Type や Scope が不適切

**問題:**
```
update(stuff): コードを更新
```

- `update` は Conventional Commits の有効な type ではない
- `stuff` は意味のない scope
- Subject が曖昧

**解決策:**

Type 選択のフローチャート:
1. 新機能追加? → `feat`
2. バグ修正? → `fix`
3. 機能変更なくコード改善? → `refactor`
4. ドキュメントのみ? → `docs`
5. テストのみ? → `test`
6. パフォーマンス改善? → `perf`
7. その他（依存関係、設定）? → `chore`

Scope は影響を受けるモジュール名を使用:
```
refactor(repository): クエリロジックを Repository に移動
```

---

### 4. 検証せずにコミット

**問題:**

コミット後にビルドエラーやテスト失敗に気づく。

```bash
git commit -m "fix(auth): JWT検証を修正"
# ... 後で気づく ...
<テスト実行コマンド>
# FAIL: 構文エラーが発生
```

**解決策:**

コミット前に必ず検証:

```bash
# 1. 型チェック / 静的解析
<型チェック・静的解析コマンド>

# 2. テスト実行
<テスト実行コマンド>

# 3. 問題なければコミット
git commit -m "fix(auth): JWT検証を修正"
```

エージェントの場合は LSP / `get_errors` 等のツールを使用して事前確認。

---

### 5. ファイルの一部だけコミットしたい

**問題:**

1つのファイルに複数の無関係な変更があり、一部のみコミットしたい。

```bash
# src/services/order-service.* に2つの変更:
# 1. 金額計算のバグ修正（完了）
# 2. 新機能の実装（まだ途中）
```

**解決策:**

インタラクティブモードで選択的にステージング:

```bash
git add -p src/services/order-service.*
```

各 Hunk（変更のかたまり）に対して:
- `y`: このHunkをステージング
- `n`: スキップ
- `s`: さらに小さく分割
- `e`: 手動で編集
- `q`: 終了

これにより、バグ修正のみを先にコミットし、新機能は後で別コミットにできます。

---

### 6. コミット後に間違いに気づく

**問題:**

コミット直後に誤字や忘れたファイルに気づく。

**解決策（まだpushしていない場合）:**

```bash
# コミットメッセージを修正
git commit --amend -m "fix(billing): 金額計算の誤差を修正"

# ファイルを追加してコミットに含める
git add forgotten_file.*
git commit --amend --no-edit
```

⚠️ **注意:** `--amend` は既にpushしたコミットには使用しないこと。

**解決策（既にpushした場合）:**

新しいコミットで修正:
```bash
git add fixed_file.*
git commit -m "fix(billing): 前回コミットの誤字を修正"
```

## If You're Unsure（判断に迷ったら）

### 変更が複数の目的を持つ

**状況:**

リファクタリングとバグ修正が同じファイルで発生。

**対応:**

ユーザーに確認:
```
この変更にはリファクタリング（関数抽出）とバグ修正（10進演算導入）の
両方が含まれています。別々のコミットに分割しますか？

オプション:
1. 分割する（推奨）- レビューしやすく、revert も容易
2. まとめる - 変更が密接に関連している場合のみ
```

**原則:** 迷ったら分割。

---

### Scope が不明

**状況:**

変更が複数のモジュールに跨っている、または小さすぎて scope が不明。

**対応:**

1. **最も影響が大きい部分** を scope にする
   ```
   feat(api): 複数エンドポイントに認証を追加
   ```

2. **全体的な変更** なら scope を省略
   ```
   chore: 依存関係を最新版に更新
   ```

3. **ユーザーに確認**
   ```
   この変更は auth, api, admin に影響します。
   最も重要な scope は何でしょうか？
   ```

---

### Breaking Change かどうか不明

**状況:**

APIの動作を変更したが、既存クライアントに影響するか不明。

**対応:**

以下を確認:

1. **エンドポイントURL変更?** → Breaking
2. **レスポンス形式変更?** → Breaking
3. **必須パラメータ追加?** → Breaking
4. **オプションパラメータ追加?** → Not Breaking
5. **内部実装のみ変更?** → Not Breaking

不明な場合はユーザーに確認:
```
この変更はAPIレスポンスに新しいフィールドを追加します。
既存のクライアントは影響を受けませんが、BREAKING CHANGE として
マークすべきでしょうか？
```

---

### コミットの粒度が不明

**状況:**

大きな機能を実装中で、どの単位でコミットすべきか不明。

**対応:**

**原則:** 1つの論理的な変更 = 1コミット

**Good（細かすぎず、粗すぎず）:**
```
1. feat(admin): 管理画面のルーティングを追加
2. feat(admin): AdminControllerの基本実装
3. feat(admin): HTTPSリダイレクト機能を実装
4. test(admin): HTTPSリダイレクトのテストを追加
```

**Bad（粗すぎる）:**
```
1. feat(admin): 管理画面のHTTPS化を完全実装
   (ルーティング、Controller、ミドルウェア、テストを全部含む)
```

**Bad（細かすぎる）:**
```
1. feat(admin): AdminController.phpを作成
2. feat(admin): index()メソッドを追加
3. feat(admin): HTTPSチェックを追加
4. feat(admin): リダイレクトロジックを追加
```

**判断基準:**
- ✅ 1つのコミットで独立してレビューできる
- ✅ ビルドとテストが通る
- ✅ 意味のある単位で説明できる

---

## チェックリスト

コミット前に確認:

- [ ] Type と Scope は適切か?
- [ ] Subject は50文字以内で WHY を含んでいるか?
- [ ] Body で変更の理由と背景を説明したか?
- [ ] Breaking Change の場合、`BREAKING CHANGE:` を明記したか?
- [ ] Issue番号を参照したか? (`Fixes #123`)
- [ ] 構文エラーやテスト失敗はないか?
- [ ] 無関係な変更が混在していないか?
