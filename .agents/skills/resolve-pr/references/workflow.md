# Detailed Workflow - resolve-pr

8段階で構成される、PRレビューコメント対応の完全なワークフロー。

## 段階1: PR番号取得

**実装:**
```
PR番号を引数から取得。無ければユーザーに質問。
例: /resolve-pr 42
```

**確認:**
- PR番号は整数（正の数）か
- 対象リポジトリで実在するPRか（`gh pr view <番号>` で確認）

---

## 段階2: PRブランチへチェックアウト

**コマンド:**
```bash
gh pr checkout <PR番号>
```

**実装ポイント:**
- チェックアウト前にステータス確認: `git status`
- 未コミットの変更があれば、ユーザーに確認・処理させる
- チェックアウト後、ローカルブランチが正しいか確認

---

## 段階3: レビューコメント取得

**コマンド:**
```bash
gh api repos/{owner}/{repo}/pulls/<PR番号>/comments \
  --jq '[.[] | select(.in_reply_to_id == null) | {id, path, line, body, user: .user.login}]'
```

**実装ポイント:**
- `in_reply_to_id == null` でトップレベルコメントのみ抽出（返信はスキップ）
- レスポンスが空配列なら「コメントなし」を報告して終了
- 取得した各コメントをメモリに保持（次段階で使用）

**出力例:**
```json
[
  {
    "id": 123456,
    "path": "app/Services/MyService.php",
    "line": 45,
    "body": "このメソッドは冗長です。簡潔にできませんか？",
    "user": "reviewer-name"
  },
  {
    "id": 123457,
    "path": "tests/Feature/MyTest.php",
    "line": 20,
    "body": "テストケースが足りません。エッジケースを追加してください。",
    "user": "reviewer-name"
  }
]
```

---

## 段階4: 未対応コメント判定

**ロジック:**
- 各コメント `id` に対して、返信を取得
- 返信がない OR 返信に「修正しました」「対応しました」等の完了表現がない → 未対応

**コマンド（コメントIDごと）:**
```bash
gh api repos/{owner}/{repo}/pulls/<PR番号>/comments \
  --jq '[.[] | select(.in_reply_to_id == <コメントID>) | {id, body, user: .user.login}]'
```

**実装ポイント:**
- 返信配列が空なら「返信なし」= 未対応
- 返信がある場合、最新の返信（idが最大）のbodyを確認
- 完了表現の判定：「修正」「対応」「改善」「更新」等のキーワード存在確認

**判定結果:**
```
未対応コメント: [id, path, line, body, user]
既対応コメント: [id, path, line, body, user, reply_body]
```

---

## 段階5: コメントごとの対応

**各未対応コメントに対して以下を実行:**

### 5-1. 妥当性判定
詳細は [decision-logic.md](decision-logic.md) を参照。

判定結果に応じて:
- ✅ 妥当 → コード修正へ
- ⚠️ 迷う → ユーザーに確認
- ❌ 妥当でない → ユーザーに説明・確認

### 5-2. コード修正（妥当な指摘の場合）
1. ファイル読込: `read_file` で指摘行周辺を確認
2. 修正内容を検討・実装
3. ファイル更新: `replace_string_in_file` で修正
4. 修正確認: `git diff` で変更を確認

### 5-3. 修正内容ログ
後で返信投稿時に使用するため、修正内容を記録:
```
コメントID123456:
- ファイル: app/Services/MyService.php
- 修正内容: メソッドを3行から1行に簡潔化
- 変更前/変更後: (記録)
```

---

## 段階6: コミット & プッシュ

**実装:**
```bash
git add <修正したファイル1> <修正したファイル2> ...
git commit -m "fix: PRレビューコメントに基づく修正"
git push
```

**コミットメッセージのパターン:**
- 単一コメント対応: `fix: 〇〇を修正（#PR番号）`
- 複数コメント対応: `fix: PRレビュー指摘に基づく複数修正`

**実装ポイント:**
- `git status` で修正ファイルを確認
- ステージング漏れがないか確認
- プッシュ前に `git log -1` で最新コミットを確認

---

## 段階7: コメントへ返信

**コマンド（コメントごと）:**
```bash
gh api repos/{owner}/{repo}/pulls/<PR番号>/comments \
  -X POST \
  -f body="修正しました。〇〇を△△に変更しました。" \
  -F in_reply_to=<元コメントID>
```

**返信文の構成:**
- 修正完了の宣言: 「修正しました」「対応しました」
- 変更内容: 「〇〇を△△に変更」「△△を削除」等
- 1-2文、簡潔に

**返信文の例:**
| 指摘内容 | 返信例 |
| --- | --- |
| メソッドが冗長 | 修正しました。冗長な処理を削除し、3行から1行に簡潔化しました。 |
| テストケース不足 | 対応しました。エッジケース3つを追加テストに含めました。 |
| 変数名の統一 | 修正しました。変数名をcamelCaseに統一しました。 |

**実装ポイント:**
- 返信投稿後は即座にAPIレスポンスで確認
- 全コメント対応後に返信を一括投稿（段階6→段階7の順序厳守）

---

## 段階8: 結果報告

**報告内容:**
```
✅ 修正完了

修正したコメント: 2件
  - ID 123456 (app/Services/MyService.php:45)
  - ID 123457 (tests/Feature/MyTest.php:20)

スキップしたコメント: 0件

コミットハッシュ: abc1234567...
プッシュ: ✅ 完了
```

**出力のパターン:**
- すべて修正済み → 成功報告のみ
- スキップがあり → スキップ理由も含める
- エラーが発生 → エラー内容と対応方法提示

---

## Error Handling

| 状況 | 対応 |
| --- | --- |
| チェックアウト失敗 | ローカル変更状況を確認・処理してから再試行 |
| API呼び出し失敗 | 認証情報・リポジトリ権限確認、エラーメッセージ表示 |
| コード修正失敗 | ユーザーに相談してから再実装 |
| プッシュ失敗 | リモート状態確認（競合等）、解決方法提示 |
