# 通知センター リニューアル PRD

## 1. 背景

現状、アプリ内の通知は「ベルアイコン → 一覧」のみで、既読/未読の管理が曖昧、
種類(システム/メッセージ/リマインド)が混在して見分けづらい、という不満が
カスタマーサポートに月平均40件寄せられている。2026年Q3のv3.0リリースに合わせ、
通知センターを刷新する。本ドキュメントは対象範囲・要件・非機能要件を定義する。

対象プラットフォームは iOS / Android / Web の3つ。本リリースではWebを優先実装し、
モバイルは後続スプリントで追従する(と思われるが、ステークホルダー間で合意が
取れているかは未確認)。

## 2. 用語

- **通知(Notification)**: ユーザーに届く単一のメッセージ。種別を持つ。
- **チャネル(Channel)**: 通知の配信経路。アプリ内 / メール / プッシュ。
- **ダイジェスト**: 一定期間の通知をまとめて1通にする配信モード。

## 3. 機能要件

通知センターは以下を満たさなければならない(MUST)。

- 通知は「種別」を持ち、システム・メッセージ・リマインドの3種別で分類されること。
- 一覧では未読が上部にまとまって表示され、未読バッジに件数を表示すること。
- ユーザーは個別通知を既読化できる。また「すべて既読」操作を提供すること(SHOULD)。
- 通知の保持期間は90日とし、それを過ぎたものは自動削除される。
- 通知設定画面で、種別 × チャネルの組み合わせごとにON/OFFを切り替えられること。

以下は将来的に対応してもよい(MAY / COULD)。

- ダイジェスト配信(日次/週次)。
- 通知のスヌーズ(指定時刻に再通知)。

なお、リアルタイム配信については「WebSocketで即時反映すること」と§3冒頭に書かれて
いる一方、§5の非機能要件では「配信遅延は最大60秒を許容」とあり、整合していない。

## 4. 配信フロー

イベント発生時の流れは概ね次の通り。サーバ側でイベントが起きると、Notification
Service がユーザーの通知設定を参照し、有効なチャネルそれぞれにメッセージを生成する。
アプリ内チャネルはWebSocket経由でクライアントにpushされ、未読カウンタが+1される。
メール/プッシュは外部プロバイダ(SendGrid / FCM)へ委譲する。配信に失敗した場合は
最大3回までリトライし、それでも失敗したらDead Letter Queueに退避する。

## 5. 非機能要件

- 通知一覧APIのp95レイテンシは300ms以下であること(MUST)。
- 配信遅延は最大60秒を許容する。
- 同時接続WebSocketは1インスタンスあたり10,000まで耐えること。
- 個人情報を含む通知本文は保存時に暗号化されていること(MUST、ローンチブロッカー)。

## 6. APIスケッチ

- `GET /api/v3/notifications?cursor=&limit=` — 通知一覧(カーソルページング)。
- `POST /api/v3/notifications/{id}/read` — 既読化。
- `POST /api/v3/notifications/read-all` — 全既読化。
- `GET /api/v3/notification-settings` / `PUT /api/v3/notification-settings`

エラーは `401`(未認証)、`403`(他人の通知へのアクセス)、`404`(存在しない)、
`429`(レート超過)を返す。
