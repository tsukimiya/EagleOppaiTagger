# PR テンプレートとメッセージ構成ガイド

Pull Request の説明文を効果的に記述するためのガイドです。

## PR 説明文の基本構成

### 標準構成（テンプレートがない場合）

```markdown
## 概要

この PR の目的と変更内容を簡潔に説明。

close #{issue番号}

## 変更内容

- 主な変更点1
- 主な変更点2
- 主な変更点3

## 動作確認

- [ ] ローカル環境で動作確認
- [ ] テストが通ることを確認
- [ ] 関連機能への影響がないことを確認

## 確認事項

- [ ] コードレビューを受けた
- [ ] ドキュメントを更新した（必要な場合）
- [ ] 既存のテストが通る
- [ ] 新しいテストを追加した（必要な場合）

## マージ条件

承認後、即時でOK

## レビューについて

1人以上のレビュー希望
```

## セクション別詳細ガイド

### 1. 概要セクション

**目的:** PR の目的を明確に伝える

**含めるべき内容:**
- 何を解決するのか
- なぜこの変更が必要か
- 関連する Issue への参照

**例（機能追加）:**
```markdown
## 概要

管理画面へのHTTPアクセスを自動的にHTTPSにリダイレクトする機能を追加します。

本番環境のロードバランサーはHTTPSのみを受け付けるため、HTTP接続時の
ユーザー体験を改善し、セキュリティを強化します。

close #122
```

**例（バグ修正）:**
```markdown
## 概要

金額計算で小数点以下が切り捨てられる問題を修正します。

浮動小数点演算による誤差が原因で、計算結果が不正確になっていました。
固定小数点（10進）演算を導入し、正確な計算を実現します。

close #234
```

**例（リファクタリング）:**
```markdown
## 概要

複雑なクエリロジックを Repository 層に移動するリファクタリングです。

Controller の肥大化を防ぎ、コードの再利用性とテスタビリティを向上させます。

close #567
```

---

### 2. 変更内容セクション

**目的:** 技術的な変更の詳細を列挙

**含めるべき内容:**
- 追加した機能・ファイル
- 変更した箇所
- 削除した内容
- 技術的な実装手法

**例:**
```markdown
## 変更内容

### 追加

- `src/middleware/force-https.*` - HTTPS強制ミドルウェア
- `infra/modules/load-balancer/` - ロードバランサー用 IaC モジュール
- `tests/https-redirect.*` - HTTPSリダイレクトのテスト

### 変更

- アプリ初期化処理 - ミドルウェアをグローバル登録
- 設定ファイル - HTTPS強制フラグを追加
- `.env.example` - 環境変数の例を更新

### 削除

- 旧HTTPチェック処理（各所に散在していたもの）
```

---

### 3. 動作確認セクション

**目的:** 変更が正しく動作することを示す

**含めるべき内容:**
- 実施した確認項目
- テスト結果
- スクリーンショット（UI変更の場合）
- 確認した環境

**例（機能追加）:**
```markdown
## 動作確認

### ローカル環境

- [x] HTTP接続時に301リダイレクトが返される
- [x] HTTPS接続時は正常にページが表示される
- [x] リダイレクトループが発生しない
- [x] 既存機能に影響がない

### テスト

```bash
<テスト実行コマンド>
# Tests: 145 passed
```

### 確認環境

- ランタイム / フレームワークのバージョン
- 実行環境（コンテナ等）
```

**例（インフラ変更）:**
```markdown
## 動作確認

### Terraform

```bash
cd <iac-dir>
terraform plan
# Plan: 8 to add, 0 to change, 0 to destroy
```

### デプロイ後確認

- [x] ロードバランサーが正常に起動
- [x] ヘルスチェックが成功
- [x] HTTPSでアクセス可能
- [x] 証明書が有効
```

---

### 4. 確認事項（チェックリスト）

**目的:** マージ前の確認漏れを防ぐ

**標準チェックリスト:**
```markdown
## 確認事項

- [ ] コードレビューを受けた
- [ ] 関連ドキュメントを更新した
- [ ] 既存のテストが通る
- [ ] 新しいテストを追加した（必要な場合）
- [ ] Breaking Change がある場合、マイグレーションガイドを追加
- [ ] 環境変数の変更がある場合、`.env.example` を更新
```

**プロジェクト固有の追加項目（例）:**
```markdown
- [ ] 依存クライアントへの影響を確認した（API変更の場合）
- [ ] 自動生成コードを再生成した（該当する場合）
- [ ] API ドキュメントを更新した
- [ ] 静的解析（型チェック / linter）のエラーがない
- [ ] フォーマッタを適用した
```

---

### 5. マージ条件セクション

**目的:** マージのタイミングや条件を明確にする

**例:**
```markdown
## マージ条件

承認後、即時でOK
```

```markdown
## マージ条件

以下を満たした後にマージ:
- 2名以上の承認
- CI/CDが全て成功
- QA環境での動作確認完了
```

```markdown
## マージ条件

本番リリース前にマージ:
- デプロイ計画確定後
- ステージング環境での検証完了
```

---

### 6. レビューについてセクション

**目的:** レビュワーへの期待を明確にする

**例:**
```markdown
## レビューについて

1人以上のレビュー希望
```

```markdown
## レビューについて

以下の観点でのレビューをお願いします:
- セキュリティ面での懸念がないか
- パフォーマンスへの影響
- エラーハンドリングの妥当性

特に @username さんのインフラ観点でのレビューをお願いします。
```

## Issue との連携

### Issue クローズの記法

```markdown
close #123
```

**その他のキーワード:**
- `close #123`, `closes #123`, `closed #123`
- `fix #123`, `fixes #123`, `fixed #123`
- `resolve #123`, `resolves #123`, `resolved #123`

複数のIssueをクローズ:
```markdown
close #123
close #456
```

または:
```markdown
close #123, #456
```

### Issue 参照（クローズしない）

```markdown
refs #789
```

## Tips

### 良い PR 説明文の特徴

1. **簡潔で明確** - 冗長な説明を避ける
2. **WHY を説明** - なぜこの変更が必要かを明記
3. **レビュワーに優しい** - 変更の意図が理解しやすい
4. **検証可能** - 動作確認の方法が明確

### 避けるべきパターン

```markdown
## 概要

コードを修正しました。
```
→ 何を、なぜ修正したかが不明

```markdown
## 概要

AdminController.phpとmain.tfを変更し、ミドルウェアを追加して、
テストを書いて、ドキュメントを更新しました。
```
→ WHAT の羅列で WHY がない

## プロジェクト固有のガイドライン

### API 変更の PR

```markdown
## 概要

ユーザー検索エンドポイントを追加します。

管理画面での効率的なユーザー検索を可能にするため、
部分一致検索と曖昧検索をサポートします。

close #456

## Breaking Changes

⚠️ なし（後方互換性を保ちます）

```

### インフラ変更の PR

```markdown
## 概要

本番環境にロードバランサーを追加します。

HTTPS通信を終端し、セキュリティを強化するため、
ロードバランサーと証明書を統合します。

close #122

## デプロイ手順

1. IaC の apply を実行
2. DNS レコードをロードバランサーに向ける
3. 旧サーバーへの直接アクセスを遮断

詳細は運用ドキュメントを参照。
```

### スキル追加・修正の PR

```markdown
## 概要

git-commit スキルを Progressive Disclosure パターンで最適化します。

コンテキストウィンドウの効率を改善するため、詳細な仕様を
references/ に分割し、必要な情報のみを段階的に読み込めるようにします。

## 変更内容

- SKILL.md を簡素化（200行 → 100行）
- references/conventional-commits.md を追加
- references/examples.md を追加
- references/troubleshooting.md を追加
```
