# Decision Logic - resolve-pr

指摘の妥当性を判定し、対応方針を決定するロジック。

## 判定フロー

```
指摘内容を読む
    ↓
妥当か判定可能か？
    ├→ YES: 妥当性判定
    │   ├→ 妥当: コード修正へ
    │   └→ 妥当でない: 対応不要（理由説明）
    └→ NO: ユーザーに相談
```

---

## 妥当性判定の基準

### ✅ 妥当な指摘（対応する）

#### パターン1: コードの品質・可読性
- 冗長な処理の削除
- 変数名・メソッド名の統一性改善
- コメント不足への指摘
- セキュリティリスク指摘
- パフォーマンス改善提案

**判定例:**
```
指摘: 「このMAPメソッドは冗長です。簡潔にできませんか？」
妥当性: ✅ 妥当（コード簡潔化）→ 修正する
```

#### パターン2: テスト・ドキュメント
- テストケース追加要求
- カバレッジ低下指摘
- API ドキュメント不足
- インテグレーションテスト追加

**判定例:**
```
指摘: 「エッジケース（負の値）のテストが足りません。」
妥当性: ✅ 妥当（テスト追加）→ テストコード追加
```

#### パターン3: ルール・規約遵守
- Linting ルール違反
- 命名規約違反
- クラス設計パターン違反
- データベーススキーマ規約違反

**判定例:**
```
指摘: 「変数名がcamelCaseで統一されていません。」
妥当性: ✅ 妥当（規約遵守）→ 名前変更
```

---

### ⚠️ 迷う指摘（ユーザーに相談）

判定基準が曖昧だと感じたら、**必ずユーザーに確認**してから対応を決定。

#### パターン1: 主観的評価
```
指摘: 「このロジックはもっと優雅に書けます。」
疑問: 「優雅」の定義が曖昧。どうリファクタリングすべき？
対応: ユーザーに確認 → 「具体的な改善案がありますか？」
```

#### パターン2: トレードオフがある
```
指摘: 「パフォーマンス重視か可読性重視か？」
疑問: 両立不可能な場合、優先度が不明
対応: ユーザーに確認 → 「プロジェクトの優先順位は？」
```

#### パターン3: 技術的判断が必要
```
指摘: 「この実装では N+1 問題が発生しているのでは？」
疑問: 実際に発生していることを確認した上での指摘か？
対応: ユーザーに相談 → 「実際にボトルネックですか？テストデータで確認しましたか？」
```

---

### ❌ 妥当でない指摘（対応不要）

#### パターン1: 仕様で既に対応済み
```
指摘: 「この変数は使われていないのでは？」
検証: コードを読むと、後続で使用されている
判定: ❌ 妥当でない（既に対応済み）
対応: 「コード行25-30で使用されています」と返信
```

#### パターン2: プロジェクト方針に反する
```
指摘: 「この処理はServices層ではなくControllers層に書くべき」
検証: プロジェクトの規約では、ビジネスロジックはServices層
判定: ❌ 妥当でない（プロジェクト規約に準拠）
対応: 「当プロジェクトではビジネスロジックはServices層です」と返信
```

#### パターン3: 誤った理解
```
指摘: 「データベースのインデックスが設定されていません」
検証: 実際にはマイグレーション内で `index()` が設定されている
判定: ❌ 妥当でない（既に実装済み）
対応: 「マイグレーション行42でインデックス設定済みです」と返信
```

#### パターン4: スキルセット外
```
指摘: 「フロントエンドのCSSをもっと整理してください」
状況: バックエンドエンジニアによるプルリクエスト
判定: ❌ 妥当でない（スコープ外）
対応: 「フロントエンドはスコープ外です。デザインチームに共有します」
```

---

## 判定チェックリスト

コメントを読んだら、以下の質問に答える：

- [ ] 指摘内容が明確に理解できるか？
  - NO → ユーザーに質問「具体的にどこを修正すべき？」
- [ ] 指摘は妥当か？（コード検証・仕様確認済み）
  - NO → 対応不要「既に実装済み・規約に準拠」と返信
  - YES → 次へ
- [ ] この修正はスコープ内か？
  - NO → 対応不要「スコープ外」と説明・返信
  - YES → 次へ
- [ ] 修正内容が決定可能か？
  - NO → ユーザーに相談「具体的な修正案がありますか？」
  - YES → コード修正へ

---

## 返信パターン

### 妥当でない場合の返信例

| 理由 | 返信例 |
| --- | --- |
| 既に実装済み | 「確認ありがとうございます。実はコード行45で既に対応済みです。」 |
| プロジェクト規約に準拠 | 「当プロジェクトではServices層にビジネスロジックを配置する規約です。」 |
| 意図的な設計 | 「パフォーマンスを優先する設計のため、この構成としています。」 |
| スコープ外 | 「フロントエンド部分はスコープ外のため、デザインチームに共有します。」 |

### 迷う場合の質問例

| 状況 | 質問 |
| --- | --- |
| 主観的評価 | 「『優雅』な実装について、具体的なリファクタリング案がありますか？」 |
| トレードオフ | 「パフォーマンスと可読性のどちらを優先すべきですか？」 |
| 技術的検証 | 「このN+1問題は実際にパフォーマンス測定で確認されましたか？」 |

---

## 実装時のポイント

1. **指摘を何度も読む**
   - 最初の読み込み時は感情的になりやすい
   - 落ち着いて、客観的に判定する

2. **仕様・規約書を確認**
   - プロジェクトドキュメント確認
   - 過去のPRレビュー例を参考に

3. **判定に自信がなければ、早めにユーザーに相談**
   - 「妥当」「妥当でない」の二者択一で迷うなら、ユーザーに任せる
   - 時間短縮のため
