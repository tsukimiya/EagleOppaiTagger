## コーディング行動指針（Karpathy Guidelines）

LLM によるよくあるコーディングミスを減らすための行動指針。  
**トレードオフ:** 速度より慎重さを優先する。些細なタスクでは判断で使い分けること。

> ⚠️ **プロジェクト固有の例外（プロジェクトの規約が優先）:**
> - プロジェクト規約として定められた**アーキテクチャ層**（レイヤード構成など）は、「単一用途の抽象化禁止」の対象外。
> - プロジェクトが是正対象としている**既存の悪習慣**は、既存スタイルに合わせず `AGENTS.md` の方針に従う。

### 1. コーディング前に考える

**仮定しない。混乱を隠さない。トレードオフを明示する。**

実装前に：
- 仮定を明示する。不確かな場合は質問する。
- 複数の解釈がある場合は提示する——黙って選ばない。
- よりシンプルなアプローチがあれば伝える。必要に応じて反論する。
- 不明点があれば立ち止まる。何が混乱しているかを明示する。質問する。

> 提示したトレードオフが「元に戻しにくい決定」を含むなら、それは ADR 候補。
> general.instructions の「意思決定の記録（ADR）」に従い、その場で申告する。

### 2. シンプルさを優先する

**問題を解決する最小限のコード。投機的な実装はしない。**

- 依頼された以上の機能は追加しない。
- 依頼されていない「柔軟性」や「設定可能性」は追加しない。
- あり得ないシナリオへのエラーハンドリングはしない。
- 200行で書いて50行にできるなら、書き直す。

「シニアエンジニアがこれを過剰に複雑と言うか？」——もしそうなら、シンプルにする。

### 3. 外科的な変更

**必要な箇所だけ変更する。自分が作ったものだけ片付ける。**

既存コードを編集する際：
- 隣接するコード・コメント・フォーマットを「改善」しない。
- 壊れていないものをリファクタリングしない。
- 無関係な不要コードに気づいたら、指摘する——削除はしない。

自分の変更が孤立コードを生んだ場合：
- 自分の変更によって不要になった import / 変数 / 関数を削除する。
- 既存の不要コードは依頼された場合を除き削除しない。

テスト基準：変更したすべての行がユーザーの依頼に直接つながること。

### 4. 目標駆動の実行

**成功基準を定義する。検証できるまでループする。**

タスクを検証可能なゴールに変換する：
- 「バリデーションを追加」→「無効な入力のテストを書き、パスさせる」
- 「バグを修正」→「再現するテストを書き、パスさせる」
- 「X をリファクタ」→「リファクタ前後でテストが通ることを確認する」

複数ステップのタスクでは、簡潔なプランを示す（SDD の TODO フローと組み合わせる）：
```
1. [ステップ] → 検証: [チェック内容]
2. [ステップ] → 検証: [チェック内容]
3. [ステップ] → 検証: [チェック内容]
```
